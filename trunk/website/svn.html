Using SVN

Introduction

Subversion (SVN) is the version management system the Libranet Basics project uses. With SVN multiple authors can work on the same source tree. This page gives a short introduction to Subversion. Please be aware that you will need a developer account to be able to use these instructions, frequent contributors will be offered an account.

The libranet-basics repository consists of a few directories. For normal usage only the trunk and branches directories are important. Trunk is used for the current development version of the book (at this moment trunk holds the book for Libranet 2.8.1). Branches will hold sources for versions of the book that are written for older Libranet versions. The 2.8.1 book will go here when Libranet 3.0 is released. If you are contributing to the cutting-edge of the book trunk/libranet-basics is the most important directory, it holds the sources of the current version of the book.

To use Subversion, you have to install the subversion package with apt-get install subversion.

Please, send a question to the lb-devel list if something is not completely clear to you after reading this page. The source repository is the heart of this project, and we'd like to keep the amount of screw-ups to the minimum ;).

Checking out the sources

The first thing you have to do is to get a source snapshot of the book. You only have to do this once, you can update this snapshot infinitely ;). Checking out the sources is really easy with the following two commands:

export SVN_SSH="ssh -l yourloginname" svn checkout svn+ssh://svn.berlios.de/svnroot/repos/libranet-basics

After executing these commands the sources will be available in libranet-basics/.

Updating your checkout

You will have to update your local checkout regularly to get the changes other people made. This can be done easily by entering the following command in the top-level directory (libranet-basics) of your checkout:

svn update

Committing changes

The most fun part is, of course, committing your own changes ;). Please make sure that:

  • you do a svn update before committing to make sure nobody else has worked on the same file,
  • you explicitly specify the files you want to commit, because SVN will commit all changed files when you don't!

Now, on to the commit command:

svn commit -m "Description of the commit." file1 file2 ... filen

Be sure to use a short, but useful description of the commit. For example:

svn commit -m "Added a section about changing process priorities." proc.xml
